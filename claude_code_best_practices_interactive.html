Best practices for Claude Code

Claude Code is an agentic coding environment — it reads files, runs commands, and autonomously works through problems. Use this guide to build intuition fast.

One constraint shapes everything: Claude's context window fills up fast, and performance degrades as it fills. Managing context is the highest-leverage skill you can develop.

Context window

The context window holds every message, file read, and command output in a session. As it fills, performance drops — Claude may miss earlier instructions or make more errors.

Context is your most important resource. Treat it like RAM — don't let it fill with irrelevant history.

Prompting

Claude can infer intent, but the more precise your instructions, the fewer corrections you'll need. Compare vague vs. specific prompts below.

Always give Claude a way to verify its work — tests, screenshots, expected outputs. This is the single highest-leverage habit.

ScenarioVague ❌Specific ✅
Testing Add tests for foo.py Write a test for foo.py covering the edge case where the user is logged out. Avoid mocks.
Bug fix The build is failing The build fails with [error]. Fix it and verify the build succeeds. Address root cause, don't suppress the error.
UI work Make the dashboard look better [paste screenshot] Implement this design. Take a screenshot of the result and compare. List differences and fix them.
New feature Add a calendar widget Look at HotDogWidget.php as an example pattern. Implement a new calendar widget that lets the user select a month and paginate forwards/backwards.

Explore → Plan → Code → Commit

Separating research from implementation avoids solving the wrong problem. Use plan mode to keep exploration separate from execution.

For small, clear tasks — a typo fix, a log line — skip the plan. Planning is most useful when the change spans multiple files or you're unfamiliar with the code.

  1. 1
    Explore plan mode
    Enter plan mode. Claude reads files and answers questions without making changes.
    read /src/auth and understand how we handle sessions and login.
  2. 2
    Plan plan mode
    Ask Claude to create a detailed implementation plan. Press Ctrl+G to open the plan in your editor.
    I want to add Google OAuth. What files need to change? What's the session flow? Create a plan.
  3. 3
    Implement default mode
    Switch out of plan mode and let Claude code, verifying against its plan.
    implement the OAuth flow from your plan. write tests for the callback handler, run the test suite and fix any failures.
  4. 4
    Commit default mode
    Ask Claude to commit with a descriptive message and create a PR.
    commit with a descriptive message and open a PR

Let Claude interview you for complex features. Start with a minimal prompt and ask Claude to use the AskUserQuestion tool — it surfaces edge cases and tradeoffs you might miss.

Configure your environment

A few setup steps make Claude Code significantly more effective across all sessions.

CLAUDE.md

Read at the start of every session. Include Bash commands, code style, and workflow rules. Run /init to generate a starter file.

✅ Include

Bash commands, code style rules that differ from defaults, testing instructions, branch naming conventions, architectural decisions, environment quirks

❌ Exclude

Things Claude can infer from code, standard conventions, detailed API docs, file-by-file descriptions, self-evident practices like "write clean code"

Permissions

Three ways to reduce approval interruptions:

  • Auto mode — a classifier handles approvals, blocking only risky actions
  • Allowlists — permit specific safe commands like npm run lint
  • Sandboxing — OS-level isolation so Claude can work freely within limits
Hooks

Hooks run scripts at specific points in Claude's workflow. Unlike CLAUDE.md instructions (advisory), hooks are deterministic — they always run.

Write a hook that runs eslint after every file edit
Write a hook that blocks writes to the migrations folder

Run /hooks to browse what's configured

Skills & subagents

Skills live in .claude/skills/ and give Claude domain knowledge or reusable workflows. Invoked automatically or with /skill-name.

Subagents live in .claude/agents/ and run in their own context with their own allowed tools — perfect for isolated research tasks.

Use a subagent to review this code for security issues

Session management

Conversations are persistent and reversible. Every prompt creates a checkpoint — you can restore conversation, code, or both at any time.

Esc
Stop Claude mid-action. Context is preserved, so you can redirect.
Esc + Esc / /rewind
Open the rewind menu — restore conversation, code, or both to any checkpoint.
/clear
Reset context entirely between unrelated tasks.
/compact
Compress history to a summary. Use /compact Focus on API changes to guide it.
/btw
Side question — answer appears in an overlay and never enters history.
--resume
Run claude --resume to pick a saved session. Use /rename to name sessions.

After two failed corrections on the same issue, run /clear and start fresh with a more specific prompt incorporating what you learned. A clean session with a better prompt almost always beats a long session with accumulated corrections.

Automate and scale

Claude Code scales horizontally. Once you're effective with one session, multiply output with parallel sessions, non-interactive mode, and fan-out patterns.

Non-interactive mode

Use claude -p "prompt" in CI, pre-commit hooks, or scripts.

claude -p "Explain what this project does" claude -p "List all API endpoints" --output-format json claude -p "Analyze this log file" --output-format stream-json
Parallel sessions

Writer / Reviewer pattern:

Session A (Writer)Session B (Reviewer)
Implement a rate limiter for our API endpoints—
—Review @src/middleware/rateLimiter.ts for edge cases and race conditions
Address review feedback: [Session B output]
Fan out across files

For large migrations, loop through a file list and invoke Claude for each:

for file in $(cat files.txt); do claude -p "Migrate $file from React to Vue. Return OK or FAIL." \ --allowedTools "Edit,Bash(git commit *)" done

Test on 2–3 files first, then run at scale.

Common failure patterns

Recognizing these early saves significant time. Each one has a reliable fix.

Anti-pattern  Kitchen sink session

You start with one task, then ask Claude something unrelated, then go back to the first task. Context fills with irrelevant information.

Fix: Use /clear between unrelated tasks.

Anti-pattern  Correction loop

Claude does something wrong, you correct it, it's still wrong, you correct again. Context becomes polluted with failed approaches.

Fix: After two failed corrections, /clear and write a better prompt using what you learned.

Anti-pattern  Bloated CLAUDE.md

If CLAUDE.md is too long, Claude ignores half of it because important rules get lost in noise.

Fix: Ruthlessly prune. If Claude already follows a rule without the instruction, delete it or convert it to a hook.

Anti-pattern  Trust-then-verify gap

Claude produces a plausible-looking implementation that doesn't handle edge cases.

Fix: Always provide verification — tests, scripts, screenshots. If you can't verify it, don't ship it.

Anti-pattern  Infinite exploration

You ask Claude to "investigate" something without scoping it. Claude reads hundreds of files, filling context.

Fix: Scope investigations narrowly, or use subagents so exploration doesn't consume your main context.
